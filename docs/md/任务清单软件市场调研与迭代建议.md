# 任务清单类软件市场调研与自有系统迭代建议

> **调研日期**：2026 年 6 月  
> **调研范围**：Todoist、Microsoft To Do、Notion、飞书任务、滴答清单/TickTick、Things 3、Any.do、Google Tasks、Apple 提醒事项、OmniFocus（参考）  
> **对标系统**：`task-planner` — 7 天综合健康与脱发管理计划任务清单（Web 端，React + Node + SQLite）  
> **目标读者**：产品 / 研发，用于指导功能迭代排期

---

## 一、自有系统定位与用户画像（迭代前提）

### 1.1 系统定位

| 维度 | 自有系统 | 主流 Todo 产品 |
|------|----------|----------------|
| **场景** | 垂直：AGA 患者按 7 日循环执行用药、运动、营养、按摩 | 通用：工作/生活任务管理 |
| **任务来源** | 预置 365 天计划种子 + 用户自定义 | 用户从零创建 |
| **核心价值** | **规范执行 + 场景演示 + 进度可视 + 多端同步** | 快速捕获 + 灵活组织 + 生态连接 |
| **协作形态** | 个人清单为主，团队邀请码（监督/结伴） | 共享项目/清单、指派、评论、甘特 |

### 1.2 目标用户

- **主用户**：遵循《7 天综合健康与脱发管理计划》的成年男性 AGA 患者  
- **次用户**：家属监督者、医生/健康管理师（只读或轻协作）  
- **使用设备**：手机浏览器为主（360px+），PC 为辅  
- **关键诉求**：「今天该做什么、怎么做、有没有漏做、坚持多久了」— 而非「管理 50 个项目」

### 1.3 已有能力基线（调研对照）

| 已有 | 缺失/薄弱 |
|------|-----------|
| 按日期任务、7 日循环、时间轴排序 | 推送提醒、重复规则自定义 |
| 任务 CRUD、勾选持久化、离线队列 | 「今日焦点」聚合视图 |
| 预计耗时标签、当日总耗时 | 专注计时 / 番茄钟闭环 |
| 场景化 Demo（米诺/非那/按摩等） | 习惯打卡与连续天数 |
| 日历月视图 + 完成进度 | 统计报表 / 趋势图 |
| JWT 登录、用户隔离、WebSocket 同步 | 团队共享清单（仅有邀请码框架） |
| 操作审计日志 | 自然语言快速录入 |

## 三、可复用功能筛选与落地方案

> 筛选原则：① 贴合 AGA 健康计划「按时执行、坚持打卡」诉求；② 可在现有 React + SQLite 栈以 **小步迭代** 实现；③ 复用已有 UI / 数据模型，避免重构成通用 PM 工具。

### 3.1 P0 — 强烈建议复用（高价值 / 低成本）

#### F1. 「今日焦点」视图（对标 MS To Do My Day / Things Today）

| 项 | 内容 |
|----|------|
| **借鉴来源** | Microsoft To Do My Day、Things 3 Today |
| **用户价值** | 打开即见「今天要完成的 8～15 项」，降低认知负担；符合移动端主场景 |
| **UI 调整** | 在 `MyTasksPage` 顶栏增加「今日 / 日历」切换；今日视图沿用现有时间轴样式，隐藏月历侧栏，顶部展示环形进度 + 总耗时（已有组件复用） |
| **流程对接** | 默认 landing 改为「今日」；数据仍走 `GET /api/tasks?date=today`；勾选逻辑不变 |
| **技术路径** | 前端纯路由/Tab 切换，**0 后端改动**；约 1～2 人日 |
| **开发成本** | ★☆☆☆☆ 极低 |
| **优先级** | **P0 — 第 1 迭代（1 周内）** |

#### F2. 任务提醒（时间型）

| 项 | 内容 |
|----|------|
| **借鉴来源** | 滴答清单、Any.do、Apple 提醒事项 |
| **用户价值** | 08:00 米诺、14:00 米诺、20:15 非那等**强时间约束**任务到点提醒，直接降低漏做率 |
| **UI 调整** | `TaskForm` 增加「提醒时间」开关；任务卡片显示 🔔 图标；设置页可选「提前 N 分钟」 |
| **流程对接** | `tasks` 表增 `remind_at`；后端定时扫描或前端 `setTimeout` + **Web Push**（二期）；一期可用 **浏览器 Notification API**（需用户授权） |
| **技术路径** | ① DB 字段 + PATCH API（0.5 天）② 前端到点检测 + Notification（1 天）③ 可选 Service Worker 后台提醒（2 天） |
| **开发成本** | ★★☆☆☆ 低～中 |
| **优先级** | **P0 — 第 1～2 迭代（2 周内）** |

#### F3. 连续完成天数 / 简单打卡统计（对标滴答习惯 streak）

| 项 | 内容 |
|----|------|
| **借鉴来源** | TickTick 习惯打卡、Todoist Karma（简化版） |
| **用户价值** | 「已连续坚持 12 天」是健康类 App 核心留存驱动力 |
| **UI 调整** | 顶栏或今日视图上方增加 streak 徽章（🔥 12）；月历格子已完成日期高亮加深（已有 progress，增强视觉） |
| **流程对接** | 新增 `user_streaks` 表或每日聚合：当日 progress=100% 则 streak+1；允许「核心任务完成即算达标」（如仅统计带 ⏱ 标签的任务） |
| **技术路径** | 后端每日汇总 API `GET /api/stats/streak`；前端展示；约 2～3 人日 |
| **开发成本** | ★★☆☆☆ 低 |
| **优先级** | **P0 — 第 2 迭代（3 周内）** |

#### F4. 子步骤 / 检查清单（对标 Things 检查项、MS To Do 步骤）

| 项 | 内容 |
|----|------|
| **借鉴来源** | Things 3、Microsoft To Do Steps |
| **用户价值** | 「米诺地尔」可拆为：确认干燥 → 量取 1mL → 分区喷涂 → 按摩 → 洗手；与现有 Demo 步骤一致，降低操作错误 |
| **UI 调整** | 任务卡片可展开 checklist；Demo 弹窗底部增加「同步为检查清单」；勾选子项不影响主任务完成，或全部勾选才自动完成 |
| **流程对接** | `task_checklist_items` 表；POST/PATCH/DELETE 子项 API；WebSocket 同步扩展 payload |
| **技术路径** | 后端 CRUD + 前端 `TaskList` 折叠组件；约 4～5 人日 |
| **开发成本** | ★★★☆☆ 中 |
| **优先级** | **P0 — 第 2 迭代（与 Demo 联动）** |

### 3.2 P1 — 建议复用（中高价值 / 中等成本）

#### F5. 专注计时 / 番茄钟（对标滴答，与 Demo 融合）

| 项 | 内容 |
|----|------|
| **借鉴来源** | 滴答清单番茄专注 |
| **用户价值** | 按摩 3 分钟、运动 30 分钟等任务可一键计时，形成「任务→执行→记录」闭环 |
| **UI 调整** | 任务卡片增加「⏱ 开始专注」；全屏翻页钟样式复用 `DemoParts.CountdownRing`；完成后写入专注记录 |
| **流程对接** | `focus_sessions` 表关联 `task_id`；可选同步 WebSocket（多端同时计时冲突时后者提示） |
| **技术路径** | 前端计时器 hook + 后端日志；**不必**首期做多端同步（滴答有，可二期）；约 5 人日 |
| **开发成本** | ★★★☆☆ 中 |
| **优先级** | **P1 — 第 3 迭代（第 4～5 周）** |

| **开发成本** | ★★★☆☆ 中 |
| **优先级** | **P1 — 第 3 迭代** |

#### F7. 家庭/监督者共享只读视图（简化版飞书协作）

| 项 | 内容 |
|----|------|
| **借鉴来源** | 飞书任务清单协作、MS To Do 列表共享 |
| **用户价值** | 家属查看「今日完成情况」无需登录主账号；医生随访时可看合规率 |

| **开发成本** | ★★★☆☆ 中 |
| **优先级** | **P1 — 第 4 迭代（第 6～7 周）** |

## 五、排除的不适配功能及原因

| 功能 | 代表产品 | 排除原因 |
|------|----------|----------|
| 甘特图 / 任务依赖 / 关键路径 | 飞书任务 | 面向项目管理，非个人健康执行；实现成本高，用户无诉求 |
| 看板 / 多维表格仪表盘 | Notion、飞书 | 任务量小（日 10～20 项），列表+时间轴足够 |
| 全功能文档/wiki | Notion | 已有独立调研 Markdown 文档，不必内置编辑器 |
| AI 语音 Ramble / 大模型助手 | Todoist | 成本高、隐私敏感（医疗场景）；一期正则 NLP 即可 |
| 6000+ 第三方集成 / Zapier | Any.do、Todoist | 小团队维护不起；健康场景集成面窄 |
| 位置提醒 | Any.do、Apple | 用药场景以**时间点**为主，位置触发价值低 |
| 四象限矩阵独立 Tab | 滴答 | 计划已按时间轴排好，再分象限增加认知负担 |
| 企业 SSO / 组织架构 | 飞书 | 当前 C 端个人用户，JWT 足够 |
| 原生 iOS/Android App | Things、滴答 | Web 移动优先 + PWA 可覆盖 80%；原生投入过大 |
| 双向 Google/Outlook 日历 | Any.do、MS To Do | 用户日历事件远少于计划内任务；单向导出即可（P2） |
| 复杂过滤器 / 自定义字段 | Notion、Todoist Pro | 分类已固定（用药/运动/营养/按摩），扩展字段需求低 |
| Karma 游戏化积分 | Todoist | 与 streak 重复；医疗场景宜「严肃+温和鼓励」而非游戏化 |

## 七、结论与行动建议

1. **不要**把自有系统做成「简版 Todoist/滴答」——主流产品的优势在**通用捕获、生态、原生 App**，短期无法追赶。  
2. **应该**在「7 日 AGA 健康计划执行器」定位上，吸收 **My Day、提醒、Streak、子步骤、番茄钟、监督视图** 等已被验证的留存功能，并与 **Demo、耗时、审计、用药规则** 深度耦合。  
3. **近期最高 ROI 三件事**：① 今日焦点视图（零后端）② 时间提醒 ③ 用药时序合规提示。  
4. **中期护城河**：场景 Demo + 检查清单 + 监督者只读 + 合规统计，形成「普通 Todo 做不到的健康执行闭环」。
